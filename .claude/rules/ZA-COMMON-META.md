# ZA-META — 治理元规则（开发期）

> **全局常驻**：本文件红线覆盖一切开发期决策与文档/治理行为，任何推进前都可能触发，
> 故**不**加 paths frontmatter。
>
> **编号**：`ZA-C-META-<NN><级>`。强制级：`*` 无自动拦截、必须自守、宁可停下问人；
> `~` 建议性、偏离须显式说明理由；条款旁 `【hook 强制】` 表示有脚本硬拦。
>
> **本文件是开发期（dev-time）治理**——约束 Claude/Terry 开发 zen-agent-mvp 本身时的行为。
> 运行期（产品内嵌 agent）治理是 `assets/system-prompt.md`（跨功能稳定基座）+
> `assets/features/<id>/`（功能规则/事实/工具面）+ `assets/skills/<fn>/SKILL.md`
> （运行时装配注入、不进开发会话）；其位置/编号/加载边界见 `ZA-C-AGENT-*`。

## ZA-C-META-01*  宗旨基准
**任何复杂度须能自证"如何让 agent 更准确地辅助用户使用宿主系统"，答不上来不引入。**
- 每个设计/实现决策都要能回答这一问；答不上的复杂度一律不引入。
- 判定：新增抽象/机制/流程无法回答该问 → 触发，砍掉或先论证。

> 反例：MVP 是模块化单体，却为"将来七系统拆分"先建 RPC 框架/消息总线，无法说明它如何提升
> 当前辅助准确度 → 投机复杂度 → 违反 META-01；端口纪律（ZA-C-WHERE-01/02）已保拆分平滑，真拆分时再做。

---

## ZA-C-META-02~  业界最佳实践优先于自创
**优先采用业界已验证模式（带证据链）；自创仅在调研确认无现成模式时允许。**
- 方案设计应优先参考顶级开源项目的核心逻辑/架构模式，而非自创。
- 自创须在调研确认无现成模式后才允许，并把依据记入 docs（人读层）。
- 偏离（选择自创）须显式说明理由。

---

## ZA-C-META-03*  执行层 ⟂ 人读层，只放规则本体
**`.claude/` 只放规则本体；设计推理/取舍/背景/出处溯源一律进 docs（人读层）。**
- AI 行为规则本体只存在于 `.claude/`；推理、取舍论证、范围声明、背景叙述不进执行层。
- 规则须自包含：判定所需事实就地写全，避免"详见 docs/X §Y"式跳转外引。
- 判定：拟在 `.claude/` 写入设计推理/取舍/背景/溯源指针 → 触发，移到 docs（人读层）。

> 反例：在某条红线下补一段"当初为何这么定"的取舍论证 → 人读内容混入执行层 →
> 违反 META-03；论证移到 docs（人读层，如 docs/reference/01-architecture.md 或 docs/adr/），红线只留结论 + 判定。

---

## ZA-C-META-04~  执行层文件尺寸守卫
**单 rules 文件 ≤200 行、单条款 ≤25 行；超了先拆再写。**
- 单分区文件超 200 行时，按需拆 `ZA-<分区>-<子>.md` 子文件（如 ZA-COMMON-HOW-PROJECT.md），各自独立适用 200 行守卫。
- 仅针对特定路径的子文件应加 paths frontmatter 按需加载；确需全局常驻须在文件头注明理由。
- 判定：拟新增后某文件 >200 行 或 某条款 >25 行 → 触发，先拆再写。
