# ZA-HOW — 编码行为准则·通用层（开发期）

> **全局常驻**：本文件是普适、跨语言、减少 LLM 通病的编码元准则，推进任何实现前都可能触发，故不加 paths。
> 语言/框架特定规范不进本文件——见同分区子文件 `ZA-COMMON-HOW-PROJECT.md`。
> 编号 `ZA-C-HOW-<NN><级>`，强制级语义见 `ZA-COMMON-META.md` 头。

---

## ZA-C-HOW-01*  实现前先思考
**动手前先说清关键假设；遇可疑前提先推回，不默默按字面实现。**
- 实现任何 task 前显式陈述所依赖的关键假设；不确定就问。
- 遇不合理/自相矛盾/前提可疑的需求先推回澄清，禁默默按字面实现。
- 判定：写实现代码前未陈述关键假设，或遇明显矛盾需求未推回 → 触发，停止先澄清。

> 反例：需求说"加缓存提速"，不问瓶颈直接加内存缓存，真实瓶颈在网络 IO →
> 未推回可疑前提 → 违反 HOW-01。

---

## ZA-C-HOW-02*  简洁优先
**解决问题的最小代码量，不写任何投机性内容。**
- 禁引入未被请求的特性；禁单一调用点的"通用"抽象；禁为不可能发生的场景写防御代码。
- 自问"资深工程师会说这过度复杂吗？"，是则简化。
- 判定：新增未被要求的功能 / 只有一处使用的抽象封装 / 为不可能输入写的分支 → 触发，删繁就简。

> 反例：只需读一个配置文件却建 ConfigProvider 工厂 + 多后端接口 → 单用途过度抽象 → 违反 HOW-02。

---

## ZA-C-HOW-03*  外科手术式改动
**只碰必须碰的；只清自己造的乱。**
- 禁顺手"优化"无关且未损坏的代码或改其既有风格；必须清理自己引入的废代码/调试残留；新代码匹配周边风格。
- 检验：每一行改动都能直接追溯到当前 task 的需求。
- 判定：diff 含与 task 无关的格式/重排/重命名，或留下自造的注释掉代码/临时日志 → 触发，回退无关改动。

> 反例：修一个 bug 顺手重排整个文件 import 和缩进 → 淹没真实改动 → 违反 HOW-03。

---

## ZA-C-HOW-04*  目标驱动执行
**先定可验证的成功标准，再写代码。**
- 实施前把 task 转成可验证的成功标准（什么算完成/怎么验证）。
- 有可测条件的改动应先确立验证手段再改代码，禁先改完再补凑验证。
- 判定：开始实施前未确立"怎样算做对"的可验证标准 → 触发，先定标准。

> 反例：接到"修复登录失败"直接改代码，没先定复现步骤/预期结果 → 无验证锚点 → 违反 HOW-04。

---

## ZA-C-HOW-05*  不伪造完成（no-silent-fallback）  【部分 hook 强制：za-bash-guard 禁 --no-verify/绕 pre-commit 门】
**禁用假实现冒充达标——空方法/TODO 占位/伪造返回/跳过校验都不算完成。**
- 禁留空方法体/`TODO`/写死的伪造返回充当真实实现；禁注释掉或跳过关键校验让流程"看起来通过"。
- 校验失败与工具报错必须如实呈现并处理，不得吞掉降级；确有未完成部分必须显式声明。
- 判定：交付物含未实现占位/伪造默认/被绕过的关键校验却报"完成" → 触发，如实声明未完成项。

> 反例：测试跑不过把断言改成 `assert True` 让它变绿并称"已修复" → 伪造完成 → 违反 HOW-05。

---

## ZA-C-HOW-06*  卡住即停并复盘
**同一路径反复失败就停下复盘，不盲目重试烧 token。**
- 同一路径连续尝试（建议 ≥3 次）仍无明显进展时，停下，输出"已尝试/阻塞点/下一步建议"。
- 禁对同一失败操作无限重复而不改策略或上报。
- 判定：同一路径 ≥3 次无进展仍盲目重试且未先复盘 → 触发，停止改为复盘。

> 反例：某命令报错反复改参数重试十几次仍不停 → 烧 token 无进展 → 违反 HOW-06。

---

## ZA-C-HOW-07*  报告如实
**测试失败就说失败并附输出；跳过的步骤明说；未验证不称完成。**
- 报告结果如实，附证据（命令 + 返回 / 测试输出 / 截图）。
- 判定：把失败/跳过/未验证说成"完成/通过" → 触发，更正为如实陈述。

> 反例：评测有 1 个场景没跑却报"全部通过" → 隐瞒跳过 → 违反 HOW-07。

---

## ZA-C-HOW-08*  注释默认不写、无时态、只述当前契约
**代码优先自描述；注释默认不加，仅在"代码无法自明"时才写，且只描述当前意图与契约（why/前提/边界/副作用/异常语义），无时态。**
- 默认不加注释：能用命名/结构表达的意图就不写注释；仅当代码确实无法自明（非显然算法、隐含前提、反直觉权衡、棘手边界）时才补。
- 注释只补代码无法表达的意图、前提、边界、副作用与异常语义；禁复述代码已自明的表面行为。
- 禁过程性注释：变更说明（"改为""移除""新增""按需求调整"）、任务/工单号、决策溯源、"详见某文档"式外引、对 reviewer/读者的喊话。
- 写注释前自问"它在 6 个月后单独读仍成立吗"——绑定到某次变更/任务/会话/外部文档的内容一律不写。
- 代码改动时同步修正受影响注释——过期注释比没有更有害。
- 判定：为自明代码加注释、或注释含变更史/任务/决策/文档引用/过程叙述/表面行为复述 → 触发，删除或改写为当前契约描述。

> 反例：`// 把 list 赋值给 result` 复述表面行为、`// 按 task-123 改成异步，原实现见旧 commit` 过程+外引+变更史 →
> 违反 HOW-08；正解：仅在必要处写 `// 异步发射避免阻塞渲染线程；失败仅记日志、不向上抛`（述意图+边界+异常语义）。
