# ZA-WHEN — deferral 与时机纪律（开发期）

> **全局常驻**：deferral 与完成判定在任何推进点都可能触发，故不加 paths。
> 编号 `ZA-C-WHEN-<NN><级>`，强制级语义见 `ZA-COMMON-META.md` 头。

---

## ZA-C-WHEN-01*  deferral 必须有界  【部分 hook 强制：za-bash-guard commit 暂存扫描拦无锚点 TODO/FIXME】
**任何 deferral 必须就地挂"具体触发锚点"；禁"以后/将来"等无锚点措辞；未了结 deferral 一律禁标 completed。**
- 触发锚点须指向可识别的未来 phase / task / 里程碑 / 事件（如"标准版 server 通道上线时"）。
- 只要 task / goal 范围内存在任一未了结 deferral，其完成状态一律禁迁；唯一放行 = 人就每条明确确认。
- 判定：产出"以后再说 / 将来考虑"式无锚点 deferral，或带未了结 deferral 仍标完成 → 触发，停止，补锚点或退回人确认。

> 反例：注释写 `// TODO 后期加指令签名` 无触发条件，且把该 task 标完成 →
> 无界 deferral + 伪完成 → 违反 WHEN-01；正解：挂锚点（如"toolgate 签发链路落地时"）+ 登记，未放行不标完成。

---

## ZA-C-WHEN-02*  验证通过才称完成
**最小验证优先，验证通过才称完成；未验证不得汇报完成。**
- 先做最小验证（命中相关子集），再按影响面扩大。
- 判定：标记完成但未跑过对应验收/验证 → 触发，先验证。

> 反例：改完装配引擎一处逻辑直接说"已修复"未跑任何用例 → 未验证称完成 →
> 违反 WHEN-02；正解：先跑命中该改动的测试/评测子集。
