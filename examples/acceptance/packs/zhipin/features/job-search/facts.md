# job-search 功能事实

> ⚠ 待核：以下页面路径、DOM 锚点与接口地址均为雏形占位，须在真实 zhipin 登录态页面核对后填实
> （触发锚点：接入真实 zhipin 页面做首轮评测时）。未核实前不得当作已知事实向用户陈述。

## 页面构成

- geek（求职者）端职位搜索/列表页（路径约 `⚠待核`，雏形 `/web/geek/job`）：顶部搜索关键词框 +
  城市 / 经验 / 薪资 / 学历等筛选控件 + 职位卡片列表。
- 每个职位卡片含：职位名 / 薪资区间 / 公司名 / 城市·经验·学历标签 / "立即沟通"入口。

## 筛选交互

- ⚠ 待核：筛选控件为原生 `<select>` 还是自定义下拉。若为自定义下拉，走 page-operate 时用 click
  命中选项行，勿用 fill 假设已选。
- ⚠ 待核：城市/薪资等是否为级联或滑块控件；如是滑块，需以 `page_snapshot` 暴露的可交互 ref 为准。

## "立即沟通"按钮（打招呼锚点）

- ⚠ 待核：职位卡片上的"立即沟通"按钮实际文案/aria-label/角色与其 ref 结构；打招呼（`job-search.greet`）
  的 click 步骤 ref 一律取自最近一次 `page_snapshot`，不写死选择器。
- ⚠ 待核：点"立即沟通"后是否跳转聊天页、跳转后 URL 形态——续作回列表用内建 `site_navigate` 时以实际
  列表页 URL 为准。

## 只读查职位接口（query-jobs）

- ⚠ 待核：`job-search.query-jobs` 的真实职位查询接口地址（tools.json 中 urlTemplate 暂占位 `/api/...`
  相对路径）、查询参数名（关键词/城市/薪资/分页）与返回 JSON 结构。未核实前只依据实际返回字段，不臆造。

## 读取

- 用 page-operate 的 `read` 步骤（带 `name` 键）采集职位卡片文本（职位名/薪资/公司/标签），回报给
  用户用于匹配判断，不臆造卡片未显示的信息。
