# SKILL：jd-match —— 岗位匹配评分方法

本篇讲**怎么评估一个 JD**：拿 `docs/profile.md` 的"我方画像"当对照基准，对每个 JD 走
"硬淘汰 → 软评分 → 回复概率 → 学历弹性 → 出评分卡"五步，给出可解释的打分与决策建议。
这里只写打分方法与可照抄的算例；"什么动作要授权、什么永拒"等红线（守）在各 feature 的 `feature.md`。

评估口径：JD 里**明确写到的**才据以判定；没写到的按"未知/中性"处理，不臆测、不替 JD 补条件。
拿不准命中与否时，宁可判为"边界待定"交用户，不武断淘汰、也不武断自动打招呼。

---

## 步骤 0：读入对照基准

从 `docs/profile.md` 取"我方"关键位：方向=AI 工程化/大模型落地/低代码；地点∈{武汉,上海,苏州,杭州}；
薪资≥35K；优势=金融银行AI·自研低代码·AI工程化落地·甲乙方交付；约束=英语弱·学历专科·年龄40。
下面每条规则都是把 JD 文本与这份基准对照的一次判断。

---

## 步骤 1：硬淘汰（Hard filter，任一命中即"弃"）

逐条比对，命中任一即停止后续打分，直接判"弃"并记一句原因。E1 学历弹性只对 H6 生效（见步骤 4）。

| 编号 | 命中条件（JD 明确写到） | 记什么原因 |
|---|---|---|
| H1 | 英语为工作语言 / 要求"英语流利" / 硬性 CET-6 及以上 | 英语弱，硬门槛不符 |
| H2 | 要求在 武汉·上海·苏州·杭州**之外**的城市坐班（远程 / 四城可选不算命中） | 地点不在可接受范围 |
| H3 | 薪资**明确**标注且上限 < 35K（"面议"/未标薪资不算命中） | 薪资低于底线 35K |
| H4 | 明确年龄上限 < 40 / "35 岁以下" / "40 岁以下"卡到 40 | 年龄超其明确上限 |
| H5 | 明显非 AI 方向（纯前端 / 纯运维 / 传统后端且无任何 AI 元素） | 方向非 AI |
| H6 | 学历硬卡"本科及以上" **且** 措辞强硬（"必须本科""硬性要求"）/ 正式大厂无弹性迹象 | 学历硬卡本科、无弹性 |

判定示例：
- "要求本科及以上，英语可作为工作语言" → 命中 H1（英语），弃。
- "base 成都" → 命中 H2，弃。
- "12-20K·15薪" → 折算月上限约 25K < 35K → 命中 H3，弃。
- "本科及以上优先"（"优先"=软措辞，非硬卡） → **不**命中 H6，进步骤 2。

---

## 步骤 2：软评分（Scorecard，算"匹配度分"0–100）

未被硬淘汰的 JD 才打分。基准分按方向契合度定，再叠加 S1–S4 命中分，封顶 100。
部分沾边只给该项一半分。

**方向基准分**
- JD 属 AI 工程化/大模型落地/低代码/Agent 工程 主航道 → 起评 **40**
- JD 沾 AI 但偏边缘（数据/算法支撑岗、AI 只是加分项） → 起评 **20**

**加分维度（命中即加，部分命中减半）**

| 编号 | 加分点（JD 需求 ↔ 我方优势） | 满命中 | 半命中 |
|---|---|---|---|
| S1 | 金融 / 银行 / 金融科技 **且** 涉 AI（银行AI、金科大模型） | +25 | +12 |
| S2 | 低代码 / 无代码 / aPaaS / 平台化建设需求 | +25 | +12 |
| S3 | AI 工程化 / 大模型落地 / RAG / Agent 工程 / 模型服务化部署 | +25 | +12 |
| S4 | 甲乙方交付 / 项目制 / 数字化项目管理经验被看重 | +10 | +5 |

`匹配度分 = min(100, 方向基准分 + ΣS命中分)`。

**分档**
- 匹配度分 ≥ 75 → 档位 **高**
- 50 ≤ 分 < 75 → 档位 **中**
- 分 < 50 → 档位 **低**

算例：
> JD：某股份制银行"金融大模型应用工程师"，做行内 RAG 知识库 + Agent 落地，含平台化沉淀。
> 方向主航道 40 + S1 满 25 + S3 满 25 + S2 半 12（平台化沾边低代码） = 102 → 封顶 **100，高档**。

> JD：某互联网大厂"AI 应用开发"，通用大模型应用，无金融、无低代码。
> 方向主航道 40 + S3 满 25 = **65，中档**。

---

## 步骤 3：回复概率（Reply odds，档位 高/中/低）

回复概率评"投了有没有戏"，与匹配度分**分开**看。起始档=**中**，按 R1–R3 逐条升降，封顶"高"、封底"低"。

| 编号 | 命中条件 | 调整 |
|---|---|---|
| R1 | 知名大厂 / 正规军 **且** 学历或年龄卡得严（专科40 在其门槛前明显劣势） | 下调一档 |
| R2 | JD 措辞正式/强硬（"必须本科""硬性""严格要求"密集出现） | 下调一档 |
| R3 | 中小 / 成长型企业 **且** 措辞弹性（"相关经验可放宽""能力优先""欢迎沟通"） | 上调一档 |

多条命中就逐档累加后再夹到 高/中/低 三档内。

算例：
> 大厂 + "本科及以上（硬性）" → R1 下调 + R2 下调，从"中"落到 **低**。
> 中小 AI 公司 + "学历不设限、能力优先" → R3 上调，从"中"升到 **高**。

---

## 步骤 4：学历弹性 E1（把 H6 从"弃"救回"边界待定"）

对应现实里"专业特别对口时企业会降学历门槛"。

- **触发条件**：该 JD 曾命中 H6（学历硬卡本科） **且** S1–S3 中至少一项为**满命中**（专业高度契合）
  **且** 企业为中小 / 成长型（非正式大厂）。
- **触发效果**：H6 由"弃"**降级为"边界待定"**——不直接淘汰，继续走完步骤 2、3 的打分，最终决策归入"边界清单交用户"。
- 若企业是正式大厂 / 无任何弹性迹象，E1 不触发，H6 维持"弃"。

算例：
> 某 30 人 AI 创业公司"低代码+大模型平台研发"，JD 写"本科及以上"但通篇"能力优先、经验突出可议"。
> 命中 H6 → 但 S2、S3 满命中 + 中小企业 → E1 触发 → H6 降级，不弃，转"边界待定"。

---

## 步骤 5：出评分卡 + 决策建议

每个 JD 汇总成一张卡，字段固定如下（可照抄此结构）：

```
{
  "company": "公司名",
  "position": "岗位名",
  "匹配度分": 88,              // 步骤2，0–100
  "匹配度档": "高",            // 高/中/低
  "回复概率档": "中",          // 步骤3，高/中/低
  "维度理由": {
    "硬约束": "未命中硬淘汰（薪资面议、地点武汉、无年龄/英语硬卡）",
    "软评分": "方向主航道40 + S1银行AI满25 + S3落地满25 = 90",
    "回复概率": "起评中；R1/R2未命中，R3未明确，维持中",
    "学历弹性": "本岗未卡学历，E1不适用"
  },
  "决策": "自动greet",
  "一句话": "银行AI+工程化落地双命中，方向极契合，直接打招呼。"
}
```

**决策规则（三选一）**：
- **自动 greet**：匹配度档=高 **且** 回复概率档≥中（中或高）。→ 交 greet 工具打招呼，成功后按 `skills/application-log` 调 `record_application` 落盘。
- **边界待定（列给用户定）**：匹配度档=中/低，或高分但回复概率=低；以及所有 **E1 触发**或 **R3 拉升但仍够不到自动线**的 JD。→ 进"边界清单"，附匹配度分+回复概率档+关键理由，交用户决定投不投。
- **弃**：命中任一硬淘汰（且未被 E1 救回）。→ 记一句原因，不打招呼、不记录。

**边界情形兜底**：无硬淘汰但软评分极低（<50 且无任一 S 命中）时，仍归"边界待定"并注明"契合度弱、建议用户斟酌"，不擅自弃、也不自动 greet。

---

## 决策速查（照此对号入座）

| 匹配度档 | 回复概率档 | 有 E1/R3 弹性 | 决策 |
|---|---|---|---|
| 高 | 中 或 高 | — | 自动 greet |
| 高 | 低 | — | 边界待定 |
| 中 | 任意 | — | 边界待定 |
| 低 | 任意 | 有 | 边界待定 |
| 低 | 任意 | 无 | 边界待定（注"契合弱，建议斟酌"） |
| —（命中硬淘汰） | — | 无 E1 救回 | 弃 + 一句原因 |

> 批量筛选时：把一屏 JD 各出一张卡，"自动 greet"的直接走打招呼流程，"边界待定"的攒成一份清单一次性给用户，
> "弃"的每条一句原因带过。这样用户看到的是"帮我投了 X 个、这 Y 个你定、这 Z 个因某某弃了"。
