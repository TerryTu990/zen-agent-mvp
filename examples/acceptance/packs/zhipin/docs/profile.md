# 用户求职画像

本篇是岗位匹配的**"我方"输入**：jd-match 规则引擎对每个 JD 打分时，以下方画像为对照基准，
评估"是否更适合我、投了是否更易获回复"。真实细节处标注"（示例，用户后补）"，请用户按实际补全后再据此匹配。

## 1. 求职方向

- **主方向**：AI 相关——AI 工程化 / 大模型落地 / AI 应用开发 / Agent 工程化。
- **匹配含义**：JD 明显非 AI 方向（如纯前端、纯运维、传统后端无 AI 元素）触发硬淘汰 H5。

## 2. 地点（硬约束）

- **可接受城市**：武汉、上海、苏州、杭州。
- **匹配含义**：要求在上述四城之外坐班触发硬淘汰 H2；远程 / 四城可选不淘汰。

## 3. 薪资（硬约束）

- **底线**：月薪 ≥ 35K。
- **匹配含义**：JD 明确标注薪资上限 < 35K 触发硬淘汰 H3；面议 / 未标不淘汰（按软评分与回复概率处理）。

## 4. 优势（软评分加分维度）

- **金融背景 + 甲乙方经验**：具备银行 / 金融科技领域 AI 项目经验，甲方与乙方两侧交付都做过。
  - 对应加分：S1（金融/银行 + AI 高加分）、S4（甲乙方/交付/项目经验加分）。
  - 细节（示例，用户后补）：参与过 ×× 银行 ×× AI 系统建设，角色 ××，成果 ××。
- **低代码平台自研 + 长期使用**：长期使用并**自研**过低代码 / 无代码 / aPaaS 平台。
  - 对应加分：S2（低代码 / 无代码 / aPaaS 需求高加分）。
  - 细节（示例，用户后补）：自研平台名 ××，支撑 ×× 业务场景，规模 ××。
- **AI 工程化落地经验丰富**：大模型落地、RAG、Agent 工程化、模型服务化部署等实战经验。
  - 对应加分：S3（AI 工程化/落地/部署高加分）。
  - 细节（示例，用户后补）：落地项目 ××，技术栈 ××，线上效果 ××。

## 5. 约束（回复概率下调 / 学历弹性触发点）

- **英语弱**：读写口语均不擅长。
  - 匹配含义：JD 要求英语为工作语言 / 流利 / 硬性 CET-6 等触发硬淘汰 H1。
- **学历专科**：最高学历为专科。
  - 匹配含义：JD 硬卡本科及以上**且**措辞强硬 / 正式大厂无弹性触发硬淘汰 H6；
    但当专业高度契合（S1–S3 强命中）且企业为中小 / 成长型时，按 E1 学历弹性**降级为"边界待定"**、不直接弃。
- **年龄 40**：
  - 匹配含义：JD 明确年龄上限 < 40 / "35 岁以下"触发硬淘汰 H4；
    知名大厂对专科 40 门槛前劣势明显，触发回复概率下调 R1。

## 6. 期望岗位类型（用于"更易获回复"排序参考）

优先级从高到低（示例，用户后补权重）：

1. **银行 / 金融科技的 AI 工程化岗**——S1+S3 双命中，最贴合优势。
2. **低代码 / aPaaS 平台的 AI 能力建设岗**——S2+S3 命中，自研经验直接背书。
3. **中小 / 成长型企业的大模型落地 / Agent 工程岗**——R3 弹性措辞、E1 学历弹性易触发，回复概率高。
4. **甲方数字化部门的 AI 项目交付 / 管理岗**——S4 甲乙方经验背书。

## 7. 使用说明

- 本画像为 jd-match 的对照输入；打分规则、硬淘汰/软评分/回复概率/学历弹性的**判定逻辑**见
  `skills/jd-match/SKILL.md`，红线（守）见各 feature 的 `feature.md`。
- "（示例，用户后补）"处请用户按真实经历补全——细节越具体，S1–S4 加分与回复概率预估越准。
