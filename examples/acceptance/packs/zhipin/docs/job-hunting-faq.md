# 求职 FAQ

本篇收录求职过程中的高频问答，作为**渐进披露**示范：平时只在 docs 索引里露出标题+摘要、不常驻占窗，
用户真问到相关问题时才由 agent 用 pack_doc 按文件名读取正文。答复为通用建议，具体判定仍以
`skills/jd-match/SKILL.md` 规则与用户画像（`docs/profile.md`）为准。

## Q1. 打招呼（点"立即沟通"）后一直没回复，怎么办？

- **先看回复概率档**：打招呼时若该岗被判为"回复概率低"（多因 R1 大厂学历/年龄卡、R2 措辞强硬），
  没回复属预期内，不必反复追。
- **优先补量而非追单**：把时间放在"匹配度高 ∧ 回复概率≥中"的新岗位上，自动打招呼覆盖面比死磕单个岗更有效。
- **边界岗位可换措辞**：对"边界待定"里用户决定投的岗，打招呼开场白突出最契合的一条优势
  （银行 AI / 低代码自研 / 大模型落地），话术见 `skills/greeting/SKILL.md`。
- **复盘用记录**：用 `list_applications` 看今天投了哪些、各自理由与决策，回头看哪类岗回复率高、据此调方向。

## Q2. 一个岗位到底该不该投？怎么判断？

判断分三步，对应 jd-match 规则：

1. **先过硬淘汰**：命中 H1（英语硬）/ H2（非四城）/ H3（薪资<35K）/ H4（年龄卡<40）/ H5（非 AI）/
   H6（本科硬卡且无弹性）任一即弃，一句话说明原因即可，不必纠结。
2. **再看软评分**：金融+AI（S1）、低代码/aPaaS（S2）、AI 工程化落地（S3）、甲乙方交付（S4）命中越多越值得投。
3. **最后看回复概率**：匹配度高 ∧ 回复概率≥中 → 直接自动打招呼；匹配度中 / 回复概率低但有弹性 → 进边界清单交用户定。

## Q3. "边界待定"的岗位是什么意思？要我做什么？

- **来源**：多是学历弹性 E1 触发的岗——专业高度契合（S1–S3 强命中）但企业硬卡本科，
  且企业为中小/成长型（有降门槛可能），故不直接弃、留给用户定。
- **你要做的**：看 agent 给的评分卡（匹配度分 + 回复概率档 + 各维度理由），
  决定"投"或"弃"。投则 agent 打招呼并记录，弃则跳过。
- **建议**：优势特别突出（如恰好是银行 AI + 低代码自研）的边界岗值得一投，专科在强专业面前企业常会松口。

## Q4. 薪资写"面议"的岗位，会被薪资硬淘汰吗？

- **不会**。硬淘汰 H3 只在 JD **明确标注**薪资上限 < 35K 时触发；"面议"/未标不淘汰。
- 想了解某岗所在行业/地区的薪资参考，可用 `job-detail.salary-benchmark`（服务端查行业薪资参考，只读）。
  注意其真实第三方接口与凭证当前为占位待核，返回值以实际接入后为准。

## Q5. 怎么查我今天投了哪些岗位？

- 用内建工具 `list_applications`（可带 `date` 参数，缺省为今天），
  它会汇总当天"投了哪些公司/岗位 + 评分 + 理由 + 决策"。
- 每次打招呼成功后，agent 会自动调 `record_application` 落盘（公司/岗位/JD摘要/评分/回复概率/理由/决策），
  故记录是自动累积的，无需手动登记。方法见 `skills/application-log/SKILL.md`。

## Q6. agent 会自动帮我改简历或自动接 offer 吗？

- **不会，这是红线**。改简历（`resume.modify-resume`）与自动接 offer（`job-detail.auto-accept-offer`）
  被声明为 forbidden（永拒）——即便对话里要求也不执行。
- **正式投递**（`job-detail.formal-apply`）为 every-call：**每一次**都需用户逐条确认后才执行，不会批量自动投。
- 只读采集简历画像（`resume.page-operate`）免授权自动进行，仅用于同步"我方"画像、不改动任何内容。
