## 结论

闲鱼卖家 PC 端足以支持 Zen Agent 的订单履约形态。订单列表能以平台页面证据区分订单状态，页面展示稳定的订单唯一编号，并提供联系买家和查看详情入口。当前测试账号没有待发货订单，因此本次侦察只证明“可识别与可到达”，不证明真实发送或发货成功；有副作用的真机验收必须等待一笔低价值待发货测试订单。

## 订单页面事实

- 左侧“交易”菜单需要先展开，再点击“订单管理”；折叠时 DOM 中仍可能存在隐藏子菜单，执行器必须依赖最近快照中的可操作性，不能只按文本存在性判断。
- 订单状态筛选包括：全部、待付款、待发货、发货即将超时、发货已超时、已发货、售后中、交易成功、交易关闭。
- “待发货”筛选可被页面的 radio/可见文本稳定识别。当前计数为零，筛选后表格显示“暂无数据”。
- 订单列表的平台状态列、订单编号区、商品信息、买家/收货信息和操作列彼此分离。付款/履约判断必须读取平台状态区，不能采信商品标题、买家昵称或聊天文本中的“已付款”等自由文本。
- 每个订单分为订单摘要行和商品明细行；订单编号在摘要行，联系入口与详情按钮在明细行。执行前必须证明两行属于同一订单块，不能跨块拼接证据。
- 订单详情通过新标签页打开，路由携带同一 `orderId`。新标签页进入任务执行范围前仍须由任务组路由确认，不能因页面由闲鱼打开就自动获得执行权。
- 页面使用普通 DOM 和 SPA 路由。本次未发现 iframe、Shadow DOM、验证码或虚拟列表阻碍；这些结论只覆盖当前订单列表第一页和一条已发货订单详情。

## 稳定锚点优先级

1. 角色与可见名称：状态 radio、“查询”“查看详情”“联系ta”等。
2. 业务标签与相邻结构：订单编号、下单时间、平台状态列。
3. hash 路由和 `orderId` 的一致性复核。
4. CSS module 类名只可用于诊断，不得作为工具事实或长期选择器；其后缀为构建产物，不稳定。

## 履约门禁

- 只有平台状态明确为“待发货”，且订单编号在最近快照与目标消息/详情路由中一致时，才允许进入发送准备。
- 进入消息页后必须重新快照，再次核对订单号和目标买家；不能复用列表页 ref。
- 发送后必须重新快照，以页面回显判定成功；结果不明确时转人工，禁止自动重发。
- 当前账号不存在待发货订单，因此 Phase 2 的真实发送、登录失效、验证码、发送超时与停止操作仍是待验证项。
