# ADR-016：确定性履约采用服务端有界自动授权

## 状态

已接受（2026-07-22）。

## 背景

`every-call` 适合自由文本发送、删除、退款等不可撤回动作，但无法支持用户预先批准后的自动发货。把“是否自动”下放插件、按自由文本任务名复用 HITL，或让模型自行声明额度，都会绕开服务端 fail-closed 决策边界。

## 决策

- 工具定义只声明 `bounded-fulfillment` 业务键映射（商品、订单、数量）；策略不进入 pack、模型上下文或客户端。
- 预批准策略只由服务端启动配置注入 toolgate，至少绑定已验签账号 `claims.hostUserId`、工具、允许商品、有效期、单订单卡密数量和每日订单上限。
- toolgate 在每次调用上先完成参数、身份、通道、DOM 围栏校验，再匹配唯一策略并原子预占订单；全部命中才从 `hitl` 提升为 `allow`。缺失、过期、歧义、超额或重复订单都回到 HITL，由人逐次确认。
- 预占状态为 `pending/completed/uncertain`。成功回执记 `completed`；失败、超时、结果 schema 不明或预占超时记 `uncertain`。三种状态都占用订单唯一键，自动路径永不重试同一订单。
- 自由文本工具继续 `every-call`；只有明确声明映射的确定性履约工具可命中策略。客户端只执行签名指令，不读取或裁决策略。

首版策略由 `ZA_FULFILLMENT_POLICIES_JSON` 注入，属于运维者预批准配置；引入持久化策略 UI 的锚点是出现多账号自助配置需求。届时只替换策略存储/审批入口，不改变 toolgate 判定语义。

## 理由

策略与额度集中在唯一决策点，保住 U7；工具只带映射、策略独立注入，避免站点 pack 与账号运营配置耦合；订单预占和 `uncertain` 语义优先防重复发货，符合履约副作用不可安全重试的事实。

## 被否方案

- 插件本地开关直接跳过 HITL：客户端承担治理判定，违反 U7。
- 继续复用 `(sessionId, task)`：任务标题由模型生成，无法表达商品、账号、有效期和额度。
- 发送失败即释放额度重试：页面结果不明确时可能已经送达，会造成重复发货。
- 将卡密或完整消息作为授权参数：会把 bearer-like 值带入模型与会话历史。

## 后果

- 正：自动履约范围可证明、可审计且超界回人工；并发调用不会突破日限额；同订单不自动重发。
- 负：MVP 策略与计数为进程内状态，服务重启后额度不延续。外置锚点是第二执行器、跨重启严格日限额或多账号并发上线；届时迁移到事务存储。
