# adr-005: 复制 zen-flux 已验证模式，不共享代码

## 状态

已接受（2026-07-04，源自架构对谈 D5）

## 背景

zen-flux-mvp 已验证一批 agent 平台机制。本项目机制层高度同构，直觉方案是抽公共库共享。但两产品演进方向不同：zen-flux 面向低代码开发流程，本项目面向宿主系统功能辅助；共享代码意味着共享演进节奏与破坏面。

## 决策

只复制**模式与契约**，不共享任何代码。搬运七件已验证资产，在本仓独立实现：

1. 装配三元组（稳定基座 + 换出块 + skillsOverride）
2. tool_call 门禁（禁 setActiveTools 路线——工具面由装配决定，不给 agent 运行时自改工具集的通道）
3. 注入自省（describeInjection 与实际注入同源，杜绝"自省说一套、注入是另一套"）
4. HITL 卡片回喂
5. events 旁路审计
6. provider 插拔白名单
7. eval 纪律

## 理由

- **演进解耦**：共享库会让任一产品的需求变更波及另一产品，版本协调成本高于重写这几件资产的成本（各自实现均在千行量级）。
- **模式已付学费，代码未必合身**：zen-flux 的实现绑定其 DSL/五相门控等专属语境，剥离这些语境的重构工作量接近重写。
- **契约同构即足够**：升级平滑靠的是契约稳定（U1-U7），不是实现共享。

## 被否方案

- **抽公共 npm 库（@zen/agent-core）**：两个消费方就抽象是过早泛化；接口被迫取两产品交集或并集，前者不够用、后者带死重。
- **fork zen-flux 改造**：继承全部不需要的 DSL/画布/门控资产，删除成本与理解成本双高，且 git 历史噪音永久存在。

## 后果

- 正：两产品演进零耦合；本项目实现只承载本项目语义，体量更小。
- 负：七件资产的 bug 修复不互通，同类缺陷可能两边各修一次。
- 负：模式漂移风险——本仓实现可能悄悄偏离已验证形态；缓解：契约进 `.schema.json` + eval 纪律看住行为面。
- 锚点：若未来出现第三个同构产品，重新评估抽库（届时有三个真实消费方）。
