# adr-002: 客户端代执行优先，服务端直调保留为标准版主通道

## 状态

已接受（2026-07-04，源自架构对谈 D2，Terry 裁定）

## 背景

API 调用协助需要一条执行通道。宿主多为 legacy / cookie 会话的内部系统：服务端直调要求宿主提供 token 化 API 或平台代持凭证；客户端代执行则在页面环境内以用户既有会话直接发请求。C1 工具定义已将通道设计为 `execution: 'client' | 'server'` 闭集（U3）。

## 决策

MVP 只实现 client 通道：服务端（③ toolgate）判定并签发一次性签名指令（nonce+ttl），插件在页面环境以用户会话执行，结果回传经服务端 resultSchema 校验后才回喂 agent。`server` 枚举值保留不删、不实现；标准版将服务端直调（token 化 API）列为首选主通道，client 降为 legacy 扩展项。

## 理由

- **零改造接入**：cookie 会话的 legacy 宿主无需提供任何 API 凭证机制即可接入，与 adr-001 的"零宿主配合"卖点一致。
- **无提权面（构造性安全）**：请求以用户浏览器会话发出，权限从构造上就等于用户本人权限——平台不持有、不代管任何宿主凭证，不存在"平台被攻破导致越权"的提权面。
- **决策仍在服务端**：通道只是执行位置，分级/HITL 判定不动（adr-003），信任模型不因通道选择而弱化。

### 关键权衡（对谈明确）：无提权面 vs 审计真实性弱一档

客户端执行换来了零提权面，代价是**审计真实性弱一档**：执行发生在不可信端，服务端无法直接观测请求与宿主的真实交互，只能校验客户端回传的结果——证据链里存在"客户端自报"环节，被篡改的插件理论上可伪造回传。裁定接受此代价，因为：其一，一次性签名指令 + ttl + resultSchema 校验已把伪造收窄到"该用户本可用自己权限做的事"，不产生超出用户权限的危害；其二，标准版以 server 直调为主通道，审计真实性在 S1 补齐；其三，MVP 阶段接入速度的价值高于审计强度增量。

## 被否方案

- **服务端直调优先**：要求宿主提供 token 化 API 或平台代持用户凭证——前者宿主多数给不出，后者制造提权面与凭证托管负担，双双违背零改造与零特权原则。
- **双通道同时实现**：MVP 复杂度翻倍，且 server 通道无真实宿主 API 可验，属投机实现。

## 后果

- 正：接入门槛最低，MVP 闭环不依赖任何宿主侧改造。
- 正：平台零特权定位（C2）从第一天成立。
- 负：审计证据链含客户端自报环节（如上权衡），S1 前审计报告需如实标注该边界。
- 负：执行受页面环境制约（CORS、页面存活期）；指令 ttl 需覆盖用户确认到执行的窗口。
