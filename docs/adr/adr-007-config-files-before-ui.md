# adr-007: 配置先文件后 UI——数据模型先于编辑界面稳定

## 状态

已接受（2026-07-04，源自架构对谈 D7）

## 背景

功能配置（⑤）由四件组成：规则 md / skills / 工具定义 / 分级矩阵。标准版有配置中心后台 UI + 版本化发布 + 评测门 + 灰度。MVP 期配置数据模型必然频繁演化（字段、粒度、组织方式都会随真实功能配置的编写经验调整）。

## 决策

MVP 配置 = git 文件即配置：`assets/features/<id>/{feature.md, facts.md, tools.json}` + `assets/skills/<fn>/SKILL.md` + `manifest.json{version, featureIdRules}`（C4）。无后台 UI。文件布局与标准版配置中心的发布产出物同构（U4），后台 UI 延后到 S2。

## 理由

- **不把不稳定模型焊死在表单里**：UI 是数据模型的凝固剂——每次模型调整都要同步改表单/校验/存储迁移，模型演化速度被 UI 拖死。文件阶段改模型只需改 schema + 迁移几个示例文件。
- **git 免费提供标准版必备能力**：版本化（commit）、不可变快照（tag/sha）、diff review、回滚——恰好是 U4 要求的快照语义，MVP 零成本获得。
- **配置作者是工程师**：MVP 期写配置的就是开发者，文件+schema 校验的编辑体验优于半成品表单。

## 被否方案

- **先做配置后台 UI**：在模型未稳定时构建 UI，每轮模型演化连带 UI 返工；且 MVP 无非工程师配置作者，UI 无真实用户。
- **数据库存配置 + 管理 API**：引入运维面与迁移负担，却不提供 git 已有的 diff/review/回滚，纯负收益。

## 后果

- 正：配置模型可以低成本快速迭代到稳定；升级=换生产端（S2 配置中心产出同构快照），装配引擎零改动。
- 正：配置变更天然过 code review 与（M4 起）评测门。
- 负：非工程师无法参与配置编写，客户自助配置能力缺位直到 S2。
- 负：featureIdRules 等运行时可热调的诉求在文件模式下需重新加载快照，接受。
