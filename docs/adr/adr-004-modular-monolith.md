# adr-004: 模块化单体起步，端口纪律使物理拆分廉价

## 状态

已接受（2026-07-04，源自架构对谈 D4）

## 背景

标准版形态是七系统独立部署，但 MVP 阶段没有多租户、没有水平扩展压力，也没有独立运维七个服务的资源。风险在于：单体若无纪律，日后拆分等于重写。

## 决策

MVP 服务端为模块化单体：一个 Node 进程，模块=pnpm workspace 包（assembly / toolgate / llm-port / audit + contracts 底座），组装唯一在 `apps/server`。以两条不变量固化拆分前提：U1（端口只传 JSON 可序列化值）、U2（模块间禁直接 import，依赖 lint 进 CI）。将来拆分时**先拆 LLM 接入层（④）**。

## 理由

- **拆分成本被前置为纪律成本**：U1 保证端口→RPC 是传输层替换而非契约重设计；U2 保证依赖图始终是星形组装，任何包可单独抽出。两条都可被 lint/类型系统自动守护，不靠自觉。
- **先拆 ④ 的依据**：LLM 接入层与业务语义耦合最低（不感知装配内容）、资源画像最独特（长连接、token 流、密钥托管、配额）、复用价值最高（可服务多产品），是收益/风险比最好的第一刀。
- 符合业界共识路径（modular monolith → 按缝拆分），避免 MVP 期承担分布式系统的全部运维复杂度。

## 被否方案

- **直接微服务**：七服务的部署/观测/联调成本压垮 MVP 期，且边界未经验证就物理固化，改边界比改包昂贵一个量级。
- **无纪律单体（快糙猛）**：短期最快，但横向 import 蔓延后拆分即重写，等于放弃标准版路径。

## 后果

- 正：MVP 单进程部署、联调简单；拆分时机与顺序保持自由。
- 正：架构约束可验证（依赖 lint、端口类型），不依赖 review 自觉。
- 负：端口纪律带来样板代码（凡跨模块必过契约类型）；单一调用点也要走端口，短期显得"过重"——接受，这正是买平滑升级的价格。
- 负：单进程内故障域不隔离（一模块 OOM 全体倒），MVP 接受，S4 解决。
