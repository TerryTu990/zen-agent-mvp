# adr-003: 决策与执行分离——判定服务端 fail-closed，通道是配置维度

## 状态

已接受（2026-07-04，源自架构对谈 D3）

## 背景

工具调用链上有两类关注点：**判定**（这个 tool_call 允不允许、要不要人确认）与**执行**（在哪里、怎么发出请求）。若二者耦合（例如客户端边执行边判定，或 agent 感知执行位置），信任边界会随实现细节漂移。

## 决策

- 判定（分级矩阵 + 身份校验 + HITL 挂起）唯一发生在服务端 ③ toolgate，fail-closed：矩阵未命中、身份不明、通道未实现，一律拒绝。
- 执行通道是工具定义（C1）里的配置维度 `execution`；agent 对通道无感知——tool_call 的形状与 observation 的形状不随通道变化。
- 客户端零治理判定：插件只渲染 HITL 卡片、按签名指令执行、回传结果，不做任何允许/拒绝判断。

## 理由

- **信任边界唯一且不动**：客户端天然不可信（可被篡改），把任何判定放进去都等于没有判定；集中在服务端才能 fail-closed。
- **通道可插拔**（U3/U7 的前提）：agent 与网关不感知执行位置，S1 增加 server 通道时只改 toolgate 内的执行器与工具配置，链路其余部分零改动。
- **HITL 语义一致**：挂起/恢复状态在服务端，卡片只是投影——换客户端形态（S3）不改 HITL 语义。

## 被否方案

- **客户端预判 + 服务端复核**：预判结果不可信则复核必须全量做，预判即冗余；且给"客户端先放行"留下心智漏洞。
- **agent 感知通道**（prompt 里区分 client/server 工具）：把部署细节泄进对话层，通道切换会改变 agent 行为面，违背装配对 agent 透明的铁律。

## 后果

- 正：安全评审面收敛到 toolgate 一处；升级拆分不移动信任边界（U7）。
- 正：工具定义可按宿主条件逐个切换通道，无代码改动。
- 负：所有 tool_call 都多一跳服务端往返，客户端无"本地快捷执行"优化空间——接受，因该优化本质是治理旁路。
