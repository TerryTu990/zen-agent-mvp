# adr-009: MVP 不做 UI 自动化替点（DOM 级填表/点按钮）

## 状态

已接受（2026-07-04，源自架构对谈 D9）

## 背景

"替用户完成操作"有两条技术路线：API 代执行（构造宿主 API 请求）与 UI 自动化（DOM 级自动填表、点按钮）。后者直觉上更通用（不需要理解宿主 API），常被期待为"agent 替我点"。

## 决策

MVP 能力面 = 引导高亮（guide-action，指给用户看）+ API 代执行（adr-002，替用户发请求）；不做 DOM 自动化替点。重新评估的锚点：标准版发布后，依据真实客户需求评估是否立项。

## 理由

- **可靠性不足以承载"替用户操作"的信任**：DOM 自动化依赖选择器对页面结构的假设，宿主前端一次改版即静默失效；失效形态还可能是"点错了地方"而非"没点"，比不做更危险。
- **主要价值已被覆盖**：用户困惑于"在哪操作"→ 引导高亮解决；用户想省去操作 → API 代执行解决且结果可校验（resultSchema）。DOM 替点的净增价值集中在"无 API 可用且流程繁琐"的窄场景。
- **治理模型不兼容**：API 代执行的每一步都过 toolgate 判定 + 签名指令 + 结果校验（U7）；DOM 操作序列难以做同等粒度的服务端判定与结果校验，会在治理面上撕开一个例外。

## 被否方案

- **MVP 内置 DOM 自动化**：为窄场景引入最不可靠的技术路线，且拖垮 fail-closed 治理模型，违背宗旨基准（无法自证"更准确"）。
- **永久排除**：需求是否真实存在应由标准版后的客户反馈裁决，现在下永久结论属过早。

## 后果

- 正：MVP 能力面全部可校验、可审计、可 fail-closed 治理。
- 正：避免选择器维护这一持续性成本黑洞。
- 负："无 API 可用"的操作场景 MVP 只能引导用户手动完成；销售侧需管理"agent 全自动操作"的预期落差。
- 锚点：标准版发布后，若真实需求集中出现，作为独立能力评估（含可靠性验收标准与治理方案），不默认沿用本决策。
