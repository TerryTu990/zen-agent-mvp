# adr-008: 讲解质量为第一验证目标，eval 纪律先于规模化

## 状态

已接受（2026-07-04，源自架构对谈 D8）

## 背景

三项能力（讲解 / 引导 / 代执行）中，讲解是使用频率最高、最先建立或摧毁用户信任的能力；而讲解质量几乎完全由功能配置内容（feature.md / facts.md / skills）决定，不由平台机制决定。平台机制再完善，配置内容错了产品就失败。

## 决策

把"讲解正确"确立为第一验证目标：eval 纪律在 MVP 期（M4）就位——评测集覆盖讲解正确 / 引导命中 / 工具触发 / HITL 触发 / 拒答边界五维度，先于任何规模化动作（多功能铺开、多租户、配置后台）引入；配置变更以评测回归为门。

## 理由

- **产品成败的主变量是内容质量**：机制层复用已验证模式（adr-005），风险已收敛；剩余最大不确定性在配置内容，验证火力应对准它。
- **eval 先行才能安全迭代配置**：配置文件会被频繁修改（adr-007 的低成本迭代），没有评测门则每次修改都在裸奔——改好一个功能可能悄悄改坏另一个。
- **拒答边界与讲解正确同权重**：agent 对配置未覆盖的问题应明确说不知道而非编造，这条边界只有 eval 能持续看住。
- 沿用 zen-flux 已验证的 eval 纪律（改 prompt/配置必跑评测集、多跑判回归），学费已付。

## 被否方案

- **先铺功能数量再补评测**：规模化会放大未被度量的质量问题，回头补 eval 时历史配置全部无基线。
- **以代执行为第一验证目标**：代执行酷但低频，且其正确性主要由工具定义与 toolgate 机制保证（可单测），不是内容质量的代表性探针。

## 后果

- 正：配置质量可回归、可比较，M4 后配置迭代有安全网；S2 配置中心的"发布评测门"直接继承该评测集。
- 负：M1-M3 期评测靠手动脚本与人工判读，节奏慢于纯手测——接受，换取基线连续性。
- 负：评测集本身成为需维护资产（功能配置变更需同步维护对应用例）。
