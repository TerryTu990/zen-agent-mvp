# adr-001: Chrome 插件优先作为 MVP 唯一客户端形态

## 状态

已接受（2026-07-04，源自架构对谈 D1）

## 背景

客户端接入层（①）在标准版有三形态：Chrome 插件 / 嵌入 SDK / 浏览器壳。MVP 只能先做一种。宿主系统分两类：可改造（能嵌 SDK）与不可改造（legacy、外采、无源码），后者在 ToB 内部系统中占比高。

## 决策

MVP 客户端仅做 Chrome 插件（`apps/extension`），实现 C3 接入层契约全部五能力；嵌入 SDK 与浏览器壳延后到标准版 S3。

## 理由

- **覆盖面最大**：插件叠加在任意页面之上，同时覆盖可改造与不可改造宿主；嵌入 SDK 只覆盖前者。
- **企业可分发**：Chrome 企业策略（force-install）使插件在 ToB 场景可集中部署，绕开个人商店安装摩擦。
- **倒逼契约自足**：插件拿不到宿主任何配合，五能力（身份获取/上下文上报/会话 UI+HITL/页面动作/代执行）必须全部自给——以最苛刻形态先行打磨 C3 契约，后续形态只会更宽松，契约不必回头改（U5）。

## 被否方案

- **嵌入 SDK 优先**：依赖宿主接入改造，排期受制于客户；且验证不了"零改造接入"这一核心卖点。
- **浏览器壳优先**：需要用户切换工作浏览器，行为迁移成本最高，不适合 MVP 验证期。
- **三形态并行**：奠基期资源不支持；且契约未经单形态验证就三处实现，返工面积三倍。

## 后果

- 正：接入层契约在最严苛条件下定型，S3 扩形态时服务端零改动。
- 负：客户环境若禁用扩展或非 Chrome 系浏览器则 MVP 无法覆盖，需等 S3 兜底。
- 负：插件与宿主页面共存的 CSP / 隔离问题需在插件内自行消化。
