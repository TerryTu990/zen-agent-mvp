# Phase 3 E2E 与加固执行记录

## 实施范围

- 新增唯一闲鱼浏览器入口 `pnpm test:e2e:xianyu`。
- E2E 加载真实 Chromium MV3 extension，并连接真实 gateway、toolgate、assembly 与 mock LLM；闲鱼页面固定在 `https://seller.goofish.com` origin，但由 Playwright route fixture 提供，不访问真实账号或订单。
- happy path 覆盖：详情页唯一“发货”单击、新快照唯一“已发货”、库存进入 `shipped-confirmed`；随后由 Chromium DevTools 强制 Service Worker 经历 `running → stopped → running`，确认不重放发货，再完成聊天页固定卡密填充与发送、新回执确认、库存进入 `sent`。
- uncertain path 覆盖：点击发货后出现未知确认弹窗且订单状态不变，只点击一次，库存进入 `manual`，不自动重试。
- E2E 对卡密 canary 和无关链接 query canary 做 LLM 原始请求、面板、会话持久化、审计四处反泄漏断言；服务端集成测试另断言卡密在原始 SSE 帧中只允许出现于签名 DOM 指令的固定 fill 值，叙事与其他帧不得复制。
- 修复空消息时闲鱼发送按钮初始 disabled 被误拒的问题：仍严格绑定输入框、发送按钮语义、页面生命周期和固定 `fill → click` 步骤；若填充后未产生唯一新回执，继续 fail-closed 转人工。
- mock LLM 的执行调用 ID 改为绑定 opaque intent，避免同会话不同订单被测试夹具错误模拟为重放。

## 发现并修复的问题

1. route fixture 最初依赖 URL fragment；HTTP 请求不携带 fragment。改为使用 hash 前的受控 `fixture` 查询参数选页。
2. E2E 最初只构建 server 与 extension，可能加载 workspace package 的旧 `dist`。改为先执行完整 `pnpm -r build`。
3. 发送按钮在消息框为空时初始 disabled，旧 toolgate 在固定 fill 之前提前拒绝。允许这一正常初态，仍以固定步骤和新增回执作为最终成功证据。
4. mock LLM 固定复用 tool call ID，导致第二个订单被防重放机制正确拒绝。调用 ID 改为绑定 opaque intent。

## 接受边界

当前结论仍是 `engineering-verified`，不是 `production-accepted`。真实账号待发货为零，未触发任何真实闲鱼副作用，也没有观察真实确认弹窗结构。生产自动扫描继续默认关闭；真实订单验收仍需用户明确指定一笔低价值待发货订单。
