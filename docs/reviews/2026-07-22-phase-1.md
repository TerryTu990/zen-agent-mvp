## 第一轮独立复审与修复

- 架构：0 P0 / 3 P1 / 1 P2。修复 prompt 与 registry/pack 同快照、版本化 compose、完整 release 回滚、飞书 guide env 名和认证目录 `0700`。
- 测试：1 P0 / 4 P1 / 4 P2。P0 为 health/副本失败处在 OR-list，`ERR` trap 不执行；改为显式激活状态机并加入失败路径行为测试。
- 强制调用 Assembly `listSites()` + `allTools()`，不再把惰性构造误当快照预检；聚合门加入真实浏览器 E2E、zip 校验、Docker runtime、非法快照拒载。
- 远端 `flock` 串行激活、快照锁、原子 deployment env/current-release，并为现有单体 compose 登记首次回滚基线。
- eval 报告修正日期并记录确定性的评测输入 SHA-256、仓库 Node/pnpm 契约与 mock 类型；避免自引用 HEAD 使每次干净验证都改脏工作树。扩展打包以 EXIT trap 恢复开发 dist。

第二轮复审发现 3 项测试/升级边界并继续修复：

- 回滚行为测试增加 `flock` 前置条件及操作日志，逐项断言 `new up → 指定故障 → old up → old validate`，消除任意前置失败也能通过的假阳性。
- 首次升级的 bootstrap 保存服务器实际旧 compose 和相对 `.env` 软链，先无副作用验证可重放，再登记 `current-release`；fixture 覆盖旧快照没有 `system-prompt.md`。
- 激活锁内创建 UID/GID 1000、`0700` 的 `/root/zen-agent/data/za`，并在容器内写读删无敏感探针，防审计/会话 fail-open 掩盖不可写挂载。
- legacy 基线登记改为与激活共用 `deploy.lock` 的 compare-and-set，测试第二个候选不能覆盖已登记基线；最终扩展 zip 先解包，再由 Chromium 直接加载做 Side Panel E2E。
- whoami 故障注入进一步暴露 Bash 函数位于 `||` 上下文时 `set -e` 不传播内部 exec 失败；数据卷、lark-cli 与 whoami 三个 smoke 均改为显式 `return 1`，分别验证触发回滚，未启用卡密时验证明确跳过 whoami。
- 最终 P2 也已关闭：legacy 发现限定 `zen-agent` compose project，并在注册锁内断言 descriptor 查询的 CID 与发现 CID 完全一致。
