## 一、目标、边界与成功标准

### 1.1 总目标

1. 把现有注入网页的对话抽屉改造成 Chrome 原生 Side Panel，使同一 Zen 会话在切换标签页和站点时保持不变。
2. 以站点包方式接入闲鱼卖家 PC 端，先闭环“识别已付款订单 → 发送卡密 → 页面复核”，同时保留上架、商品管理等能力的同构扩展入口。
3. 在闲鱼闭环稳定后，接入一个飞书多维表格完成卡密领取、发送前预占和发送后回填，不建设专用运营后台。

### 1.2 关键前提

- 闲鱼卖家 PC 端入口为 `https://seller.goofish.com/`，用户已在真实 Chrome 中登录。
- 全自动的运行边界是：Chrome 正在运行、闲鱼登录有效、Zen 工作标签页可访问；Chrome 关闭后不承诺继续执行。
- 闲鱼首版以可见 DOM 操作为主。客户端 API 只有在 Phase 2 实测确认稳定且边界可接受时才作为可选工具；无可用官方接口时不启用服务端闲鱼 API。
- 页面付款状态和订单号必须来自闲鱼平台页面证据，不能采信买家自由文本。
- 卡密原文不得进入模型上下文、对话历史、审计、截图、错误信息或仓库。

### 1.3 不在本轮实施范围

- 移动端、云手机或桌面常驻执行器。
- 闲鱼私有协议逆向、风控绕过、验证码绕过。
- 多账号高并发、复杂库存、报表、运营后台。
- 飞书多表业务模型或 Zen 专用飞书管理页面。
- 未经实测就一次性配置闲鱼全部功能。

```text
第一步：Chrome 插件形态改造
  Side Panel + 稳定会话 + 页面执行路由 + 执行偏好
                         ↓ 验证门 1
第二步：闲鱼站点能力
  真机侦察 + site pack + facts/rules/skills/tools + 单订单发货闭环
                         ↓ 验证门 2
第三步：飞书卡密集成
```

## 三、第一步：完成 Chrome 插件形态改造

### 3.1 产品行为

- 点击 Zen 工具栏图标打开 Chrome 原生 Side Panel。
- Side Panel 承载对话、HITL、工具进度、当前任务、停止按钮和执行偏好。
- 刷新闲鱼页面、切换同组标签页或跨站导航时，对话不重建、不丢失、不镜像出多个 UI。
- content script 不再渲染对话抽屉，只负责上下文采集、页面快照、页面高亮和签名指令执行。
- Side Panel 绑定一个明确的 Zen 任务会话；用户切到无关标签页时只更新“当前浏览页”提示，不自动授予该页执行资格。
- 现有标签组继续作为执行范围；跨站任务仍通过同一组延续会话。

### 3.2 架构与文件级任务

#### 批次 1A：Side Panel 壳

- `apps/extension/manifest.json`
  - 增加 `side_panel` 页面声明和 `sidePanel` 权限。
  - 保持 Manifest V3；工具栏 action 改为打开 Side Panel。
- 新增 `apps/extension/sidepanel.html` 与对应 TypeScript 入口。
- 调整扩展构建脚本，使 side panel 入口与 HTML 一并进入发布产物。
- 扩充 `chrome.d.ts` 的最小 Side Panel 类型，不引入无关 Chrome API。

#### 批次 1B：拆分 UI 与页面执行器

- 从 `content.ts` 移除固定定位抽屉和对话生命周期。
- 将 `conversation-hitl.ts` 的渲染能力迁到 Side Panel；markdown、工具卡、HITL 卡复用现有实现。
- content script 保留：`context-report`、`page_snapshot`、`dom-steps`、`page-action`、`delegated-execution`。
- background 成为唯一会话桥：一个会话一条 SSE，不因多个 content script 或 Side Panel 重连而重复订阅。
- 下行路由调整为：
  - 叙事、状态、工具卡、HITL → Side Panel；
  - snapshot、guide、exec-instruction → 当前任务组的权威执行页；
  - 执行结果仍由 background 盖章 sessionId 后上送。

#### 批次 1C：会话持续与恢复

- Side Panel 打开时恢复当前任务组映射的 sessionId。
- service worker 被回收、Side Panel 重开、页面刷新后复用服务端会话，不重复执行 nonce 指令。
- Side Panel 明示当前任务组、执行页和站点；没有可用执行页时 fail-closed，仅允许对话和诊断。
- 保留“停止”语义：停止页面批次，并吊销服务端任务级授权。

#### 批次 1D：执行偏好

Side Panel 提供用户偏好，而不是把治理决策下放客户端：

- 自动选择
- 仅客户端 DOM
- 优先客户端 API
- 优先服务端 API

偏好通过 additive 契约上报服务端；服务端只在当前站点包已经声明、身份有效且策略允许的工具集合内选择。不可用通道必须提示并暂停，不做未声明的静默降级。客户端不读取 `riskTier`、不自行决定 HITL。

### 3.3 必要设计决策

Phase 1A 开工时，用下一个可用 ADR 编号记录 Side Panel 决策，至少说明：

- Side Panel 是 Chrome 客户端内部实现，不改变 C3 五能力契约（U5）。
- 对话作用域与页面执行作用域分离。
- Side Panel 全局持续不等于全部标签页均可执行。
- 原网页抽屉完全退出，不保留两套 UI。

### 3.4 验证门 1

自动验证：

- extension 单测覆盖 Side Panel ↔ background ↔ content 三方路由。
- 覆盖 service worker 重启、Side Panel 重开、content 重连、多个标签页时的单 SSE 和单次执行。
- C3 schema、nonce/ttl、resultSchema 和 toolgate 测试不得回归。
- `pnpm lint:deps`、`pnpm -r build`、`pnpm -r --workspace-concurrency=1 test` 全绿。

真机验收：

1. 在站点 A 打开 Zen 并发送一轮对话。
2. 切换到闲鱼和其他站点，对话内容保持不变。
3. 刷新或关闭当前页面再恢复，Side Panel 不重复消息、不重复执行。
4. 在任务组外打开页面，Zen 不对该页执行动作。
5. 点击停止后，正在运行的 DOM 批次终止，后续已授权调用不再自动放行。

回滚点：批次 1A 至 1D 各自保持可独立回退；进入验证门 1 前不删除现有抽屉实现，验证通过的同一提交再完成旧 UI 清理，最终产物只保留 Side Panel。

## 四、第二步：完成闲鱼网站操作与站点能力

### 4.1 站点包定位

新增一个 `xianyu-seller` pack。它是 Zen Agent 的普通站点能力包，不在 extension、gateway 或 toolgate 中写闲鱼特判。

首个生产闭环只承诺：

1. 定位卖家中心及订单页面。
2. 读取平台展示的订单号、商品、买家脱敏标识和付款状态。
3. 打开订单对应的发货或消息入口。
4. 发送指定的测试内容，随后发送卡密占位引用。
5. 重新观察页面并验证发送结果。
6. 页面支持时完成发货状态更新并再次复核。

上架、编辑、下架、改价等功能统一收进同一 pack；只有在对应页面完成真机侦察和独立评测后才启用工具，不预写未经验证的 facts。

### 4.2 批次 2A：只读真机侦察

使用用户已登录的 Chrome，从以下入口开始：

`https://seller.goofish.com/?site=COMMONPRO&spm=a21107h.42829791.0.0#/seller-data/data`

先只读观察，不触发上架、发送、发货、删除等副作用：

- 记录 SPA hash 路由与实际 feature 映射。
- 确认“我卖出的”、订单列表、订单详情、聊天、发货入口是否均可从 PC 端到达。
- 确认付款状态的稳定可见文本、角色或结构。
- 确认订单唯一标识是否可见且在页面切换后保持稳定。
- 检查弹窗、iframe、虚拟列表、Shadow DOM、自绘组件和验证码形态。
- 为每个页面整理稳定锚点，优先 aria-label、role、可见文本和业务字段，禁止依赖动态 id。
- 不把含买家信息、订单内容或卡密的截图提交进仓库。

若付款状态或订单号无法形成可靠页面证据，Phase 2 在“只读订单辅助”处停止，不进入自动发送。

### 4.3 批次 2B：建立 pack、feature 与技能

在既有 registry/pack 布局中登记 `xianyu-seller`，第一批 feature 建议为：

- `xianyu-seller-data`：卖家中心数据与导航。
- `xianyu-orders`：已售订单发现、读取和状态判断。
- `xianyu-fulfillment`：进入消息/发货入口、发送和页面复核。
- `xianyu-listings`：仅登记已完成侦察的上架与商品管理能力。

每个 feature 配置：

- `feature.md`：仅放 `ZA-FEAT-*` 运行规则与边界。
- `facts.md`：只写真机确认的页面事实和交互注意事项。
- `tools.json`：声明 DOM、客户端 API 或服务端 API 工具；首版默认 DOM。
- `skills/`：编排“观察 → 判断 → 操作 → 再观察”的操作技法，不在 skill 中重复安全规则。
- `eval/scenarios.json`：覆盖讲解、引导、工具、HITL、拒答五维度，且不复用配置内示例。

### 4.4 批次 2C：单订单、无飞书闭环

在接飞书前，用人工提供的一条测试卡密占位值完成单订单流程：

```text
page_snapshot
  → 找到已付款订单及唯一订单号
  → 展示任务计划并获得授权
  → 进入订单消息或发货入口
  → 在发送前再次核对订单号与付款状态
  → 发送测试内容
  → 重新 page_snapshot
  → 以页面证据判定成功、失败或不明确
```

测试卡密只在当前人工验收输入中使用，不进入仓库、fixture、日志和评测素材。

### 4.5 批次 2D：自动履约授权

现有“对外发送每次确认”策略不能直接满足自动发货。实现前新增一个 ADR，明确区分：

- 任意自由文本、删除、改价、退款等操作继续 `every-call`。
- 确定性自动履约可在用户预先批准的有界策略内自动执行。

首版有界策略至少包含账号、允许商品、有效期、单次卡密数量和每日订单上限。每个订单仍由服务端 toolgate 逐次验证；超出策略、页面状态不明、登录失效或验证码出现时 fail-closed 并转人工。

### 4.6 验证门 2

- URL→pack→feature 装配准确，hash 路由切换后工具面同步换出。
- 非闲鱼 origin、未登记路径和未知页面不能调用闲鱼工具。
- DOM ref 必须来自最近快照；每次操作后必须重新快照复核。
- 买家文本“已付款”不能触发发货，必须匹配平台付款状态证据。
- 发送结果不明确时不能自动重复发送。
- 闲鱼 pack 五维评测至少重复 3 跑无回归。
- 全仓 build、串行 test、dependency lint 和命中 pack 的 eval 全绿。
- 真机用低价值测试订单分别验证成功、登录失效、元素变化、验证码/弹窗、发送超时和停止操作。

通过条件：不依赖飞书，也能由人工输入一条测试值，稳定完成一个订单的“识别 → 发送 → 页面复核”。

## 五、第三步：集成飞书卡密领取与回填

### 5.1 飞书接入纪律

飞书操作和 token 接入以 `/Users/terrytu/Workspace2025/Working/feishu/AGENTS.md` 及该项目现有对外文档/规划为准：

- 共用飞书应用 `agent-general`。
- CLI 固定显式使用 `--profile general`，个人资源默认 `--as user`。
- 任何重要写入前先执行 `lark-cli --profile general whoami`。
- Phase 3A 开工时先读取“Token 中转｜项目导航”“Token 中转服务｜对外说明（草稿）”及相关技术规划，复用既有 token 获取和刷新边界，不在 Zen 内另造飞书 OAuth/token 存储。
- 不开启外部分享，不修改 owner，不覆盖或删除现有资源。
- 管理员可用飞书 UI、MCP 或 CLI 维护卡密；Zen 运行时只消费经过确认的最小接口。

### 5.2 单张卡密表

仅创建或登记一张多维表格，字段最小化为：

| 字段 | 语义 |
|---|---|
| `card_id` | 卡密记录唯一编号 |
| `product_key` | 闲鱼商品 ID、SKU 或约定商品标识 |
| `card_secret` | 卡密原文，仅执行连接器可读取 |
| `status` | `available` / `reserved` / `sent` / `manual` |
| `order_id` | 预占或使用该卡密的闲鱼订单号 |
| `updated_at` | 最近更新时间 |
| `note` | 可选异常说明，不写卡密原文 |

创建前按精确标题搜索；如果既有表满足字段要求，直接复用并把链接登记到飞书项目导航，避免重复资源。该表保持私有，不作为模型知识库。

### 5.3 批次 3A：最小飞书连接器

连接器只实现四个动作：

1. 按 `order_id` 查询已有关联记录。
2. 按 `product_key` 获取一条 `available` 卡密。
3. 发送前写入 `reserved + order_id + updated_at`。
4. 页面确认成功后写入 `sent`；结果不明确时写入 `manual`。

连接器通过 JSON 可序列化端口接入，并由 `apps/server` 组装；不得让 assembly、toolgate 或 extension 横向 import 飞书实现。token 由既有 token 业务在运行时提供，不进入工具参数、模型或日志。

### 5.4 批次 3B：订单幂等流程

```text
发现已付款订单
  → 先按 order_id 查询
  → 已有关联：复用同一 card_id
  → 没有关联：领取 available 记录
  → 先回填 reserved + order_id
  → 再向闲鱼发送 card_secret
  → 页面复核成功：回填 sent
  → 结果不明确：回填 manual，停止自动重试
```

首版由单个 Zen 执行器串行处理订单；不引入额外 SQLite 或复杂分布式锁。出现第二个并发执行器或多账号并行需求时，以“同一张卡密被两个订单竞争”的实测风险为触发锚点，再设计飞书条件更新或独立事务账本。

### 5.5 批次 3C：周期扫描与小流量全自动

- 使用 `chrome.alarms` 周期唤醒并维护一个明确的闲鱼工作标签页。
- 浏览器、登录、目标页面或预授权任一条件不满足时暂停，不自动新建不可控页面。
- 每轮按串行方式处理少量订单；频率和批量上限由服务端策略限制。
- Side Panel 展示运行中、已暂停、缺货和人工核对状态，并提供立即停止。
- 卡密不足、飞书访问失败、回填失败、页面结果不明确均不继续下一个有副作用步骤。

### 5.6 验证门 3

至少覆盖：

- 正常预占、发送、回填 `sent`。
- 同一订单重复扫描只复用原 `card_id`。
- 发送前崩溃后从 `reserved` 恢复，不领取第二条卡密。
- 发送后响应超时转 `manual`，不自动重发。
- 飞书领取或预占失败时闲鱼侧不发送。
- 闲鱼成功但飞书最终回填失败时停止后续处理并提示人工恢复。
- 模型请求、对话历史、审计、终端输出和测试结果中无 `card_secret`。
- 飞书写入依据 CLI/API 的成功状态验证，并回读目标记录确认字段正确。
- 全仓 build、串行 test、dependency lint、闲鱼 pack 三跑 eval 和真机小额验收全部通过。

通过条件：单账号、单工作标签页、小批量订单连续运行时，无重复领取、无重复发送、无卡密泄露；所有不确定状态均暂停并可人工定位。

## 六、提交、发布与回滚纪律

建议按以下提交边界推进：
