# Zen Commerce Agent 首期实施方案

> 状态：实施中。首期只交付闲鱼卡密订单自动履约；客服、商品发布和经营分析保留有触发锚点的后续规划，不创建空实现。
>
> 生产环境：`ssh lingm2`，目录 `/root/zen-agent`，对外入口 `https://agent.flash-api.com`。现有通用站点快照由本方案的电商生产快照替换。

## 1. 产品与仓库定位

当前仓库直接演进为 **Zen Commerce Agent**，不复制通用底座、不新建第二个代码仓库。产品首个渠道为闲鱼卖家 PC 端 `https://seller.goofish.com/`，飞书作为卡密库存账本；运行时继续复用现有 Assembly、Toolgate、LLM、Audit、Chrome 接入层和模块化单体。

内部包名暂时保留 `@zen-agent/*`，避免与业务无关的大规模重命名。产品名、生产快照、扩展展示名与运维文档先完成电商化；首期真实验收通过后，再以发布里程碑决定是否重命名仓库目录与包 namespace。

## 2. 首期范围

首期闭环：

```text
周期触发
→ 识别待发货订单及平台证据
→ Toolgate 预授权
→ 飞书按订单查重并预占卡密
→ 闲鱼聊天页发送固定卡密通知
→ 页面回执确认消息已接纳
→ 更新闲鱼订单发货状态
→ 页面证据确认订单状态
→ 飞书回填 sent
```

### 2.1 成功定义

一次自动履约只有同时满足以下条件才记为 `completed`：

1. 订单号、商品和待发货状态来自同一闲鱼订单块或详情页平台字段。
2. 飞书以规范化订单号完成查重和单卡预占。
3. 卡密消息回执数量恰好增加 1，且最新状态属于允许枚举。
4. 闲鱼订单状态经发送后页面重新观察，明确变为已发货。
5. 飞书目标卡记录写入 `sent` 并按 `card_id` 回读一致。

任何一步失败、不唯一、超时、换页、验证码、登录失效或结果不明确均进入人工路径；已经发生外部副作用的订单不得自动重发卡密。

### 2.2 默认执行顺序

默认采用“预占卡密 → 发送并验证卡密消息 → 更新并验证订单状态 → 飞书终态回填”。这样可以避免订单先显示已发货但买家未收到卡密。若闲鱼真实页面强制先更新订单状态，必须在只读侦察中取得证据，再调整状态机并增加对应故障夹具；不得仅按按钮名称推断。

### 2.3 首期不做

- 不绕过验证码、登录、风控或平台限制。
- 不逆向闲鱼私有协议。
- 不建设通用飞书 SDK、OAuth 服务或运营后台。
- 不实现多账号并发、多执行器或复杂分布式锁。
- 不实现客服、商品发布、经营分析代码。
- 不用真实订单做自动化测试；生产自动扫描默认关闭。

## 3. 架构落点

| `packages/toolgate` | 服务端 fail-closed 判定、额度、幂等、签名、nonce/ttl、结果校验 | 对话生成、库存读写 |
| `packages/card-inventory` | 飞书 Base 查重、预占、attempt 标记、sent/manual 回填与写后回读 | 页面操作、模型调用、并发事务伪装 |
| `packages/fulfillment` | 预授权后领取库存、登记 opaque intent、协调消息与订单终态 | DOM 执行、凭证管理 |

## 6. 发货状态实现策略

发货状态更新作为与卡密消息不同的受控副作用建模：

- 消息发送继续使用现有 bounded fulfillment intent 和回执证据。
- 订单状态更新使用由服务端从已完成消息意图派生的 opaque intent；模型不能提交订单号、按钮 ref、物流方式或自由 DOM steps。
- intent 绑定账号、订单、商品、订单详情 URL、页面实例、允许动作、消息完成凭据和绝对有效期。
- Toolgate 只签发真机侦察确认过的固定步骤；结果仍需重新快照验证平台状态。
- 同订单状态动作至多签发一次。超时或结果不明确标记 `manual`，不得自动再点。
- 发货按钮引发二次确认、物流选择或协议确认时，全部步骤必须来自受控夹具和真机观察，不允许模型临场扩展。

## 7. Harness 与 E2E

### 7.1 自动验证层次

1. 单元测试：飞书状态迁移、订单/消息 intent 状态机、幂等、错误脱敏。
2. 服务端集成：预授权必须先于库存访问；消息未完成不得签发订单状态动作；结果不明确进入人工闩锁。
3. AI Harness：讲解、引导、工具触发、HITL、拒答五维度每场景至少 3 跑。
4. 浏览器 E2E：真实扩展 + 受控闲鱼夹具，覆盖订单筛选、聊天发送、回执、发货确认、状态复核。
5. 生产冒烟：仅健康检查、快照装载与无副作用页面识别；真实低价值订单需用户单独授权。

### 7.2 必测故障

- 待发货计数为零。
- 买家文本伪造“已付款”。
- 订单号与状态不在同一订单块。
- 飞书库存不足、查询失败、写入失败、写后回读不一致。
- 消息发送超时、回执未增加、回执增加超过 1。
- 消息成功后切页或页面实例变化。
- 发货确认弹窗结构变化、验证码、登录失效。
- 发货点击成功但订单状态未变化。
- Service Worker 在任一步骤间重启。
- 重复 alarm、重复 runId、重复 tool call 与迟到回执。

## 8. 阶段、提交与独立复审

### Phase 0：产品定型与计划

- 分支改为 `codex/zen-commerce-agent`。
- 本文成为首期实施锚点。
- 验收：范围、成功语义、延期锚点与生产边界无歧义。

### Phase 1：生产快照与飞书收口

- 闲鱼 pack 提升为正式 `assets` 快照。
- 产品展示与部署说明电商化。
- 飞书运维规则入仓；镜像具备明确的 `lark-cli` 运行依赖。
- 验收：Assembly 加载生产快照；功能评测、构建、单测通过。

### Phase 2：订单发货状态闭环

- 只读侦察真实发货页面。
- 实现服务端可信派生、固定动作、状态复核和人工闩锁。
- 验收：受控夹具完成消息与订单双证据闭环；所有异常不重试。

### Phase 3：E2E 与加固

- 扩充 Harness、浏览器 E2E、SW 恢复和回归用例。
- 验收：全仓 build/test、依赖 lint、功能评测和浏览器 E2E 全绿。

每个 Phase 完成后先提交，再分别由独立子 agent 从架构边界和测试/故障覆盖两个方向审查；主流程修复确认的问题并补提交，审查未闭环不得进入下一 Phase。

### Phase 4：发布

- 构建已提交 HEAD 的 linux/amd64 镜像与 Chrome 插件 zip。
- 部署服务端与 `assets` 快照到 `lingm2:/root/zen-agent`。
- 验证服务器本机及 `https://agent.flash-api.com/healthz`。
- 生产自动扫描保持关闭；真实订单验收等待用户明确指定低价值待发货订单。
