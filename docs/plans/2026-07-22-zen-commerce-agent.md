# Zen Commerce Agent 首期实施方案

> 状态：实施中。首期只交付闲鱼卡密订单自动履约；客服、商品发布和经营分析保留有触发锚点的后续规划，不创建空实现。
>
> 生产环境：`ssh lingm2`，目录 `/root/zen-agent`，对外入口 `https://agent.flash-api.com`。现有通用站点快照由本方案的电商生产快照替换。

## 1. 产品与仓库定位

当前仓库直接演进为 **Zen Commerce Agent**，不复制通用底座、不新建第二个代码仓库。产品首个渠道为闲鱼卖家 PC 端 `https://seller.goofish.com/`，飞书作为卡密库存账本；运行时继续复用现有 Assembly、Toolgate、LLM、Audit、Chrome 接入层和模块化单体。

内部包名暂时保留 `@zen-agent/*`，避免与业务无关的大规模重命名。产品名、生产快照、扩展展示名与运维文档先完成电商化；首期真实验收通过后，再以发布里程碑决定是否重命名仓库目录与包 namespace。

## 2. 首期范围

首期闭环：

```text
周期触发
→ 识别待发货订单及平台证据
→ Toolgate 预授权
→ 飞书按订单查重并预占卡密
→ 更新闲鱼订单发货状态
→ 页面证据确认订单状态
→ 闲鱼聊天页发送固定卡密通知
→ 页面回执确认消息已接纳
→ 飞书回填 sent
```

### 2.1 成功定义

一次自动履约只有同时满足以下条件才记为 `completed`：

1. 订单号、商品和待发货状态来自同一闲鱼订单块或详情页平台字段。
2. 飞书以规范化订单号完成查重和单卡预占。
3. 同一订单从“待发货”前态经过本次签名动作后，重新观察为“已发货”；旧状态、错误订单或人工并发变化均不能归因于本次动作。
4. 前后消息证据绑定同一账号、订单、买家会话、URL 和页面实例；新增回执为卖家方向，在 intent 有效窗口内唯一增加且状态为“未读”或“已读”。优先绑定页面稳定回执键；页面没有稳定键时，必须保持单飞并把任何竞争消息变化判为不明确。卡密正文不进入快照或关联键。
5. 飞书目标卡记录写入 `sent` 并按 `card_id` 回读一致。

任何一步失败、不唯一、超时、换页、验证码、登录失效或结果不明确均进入人工路径；已经发生外部副作用的订单不得自动重发卡密。

### 2.2 执行顺序

按业务要求采用“预占卡密 → 更新并验证订单状态 → 发送并验证卡密消息 → 飞书终态回填”。Phase 2 真机只读侦察只确认可安全到达的页面、按钮、字段和最终不可逆确认之前的弹窗；完整动作与终态证据先由受控夹具验证。真实终态只在用户授权的一笔低价值待发货订单上验证并进入 `production-accepted`；若真实弹窗只能在不可逆点击后出现，也归入该授权验收。若页面证据表明顺序不可执行，不得自行反转，须暂停并由用户裁决。

订单状态已经更新但卡密消息失败或不明确时，记录持久人工闩锁并停止，不自动重发消息、不处理同商品下一单；人工依据已预占卡密和订单记录完成对账。

### 2.3 首期不做

- 不绕过验证码、登录、风控或平台限制。
- 不逆向闲鱼私有协议。
- 不建设通用飞书 SDK、OAuth 服务或运营后台。
- 不实现多账号并发、多执行器或复杂分布式锁；生产只允许单服务副本和单飞履约 run，启动/部署检查不满足时自动扫描保持关闭。
- 不实现客服、商品发布、经营分析代码。
- 不用真实订单做自动化测试；生产自动扫描默认关闭。

## 6. 发货状态实现策略

发货状态更新作为发生在卡密消息之前、但与消息不同的受控副作用建模：

- 首次扫描时由可信订单页证据保存规范订单号、商品、待发货前态和订单详情 URL；模型不能提交或改写这些业务键。
- 进入订单详情或发货页后必须取得新的 `pageInstanceId` 和快照，重新绑定同一订单号、商品与待发货状态；发货控件 ref 只取自这份新快照。
- 订单状态更新使用服务端派生的 opaque intent；模型不能提交订单号、按钮 ref、物流方式或自由 DOM steps。
- intent 绑定账号、订单、商品、详情 URL、页面实例、允许动作、前态证据和绝对有效期。
- Toolgate 只签发真机侦察确认过的固定步骤；结果仍需重新快照验证平台状态。
- 同订单状态动作至多签发一次。超时或结果不明确标记 `manual`，不得自动再点。
- 发货按钮引发二次确认、物流选择或协议确认时，全部步骤必须来自受控夹具和真机观察，不允许模型临场扩展。
- 订单状态明确变为已发货后，服务端才可进入现有消息 intent 的准备与发送；消息失败不回滚订单状态，转持久人工处理。

### 6.1 持久化状态与恢复

首期复用飞书记录的 `status + note` 做最小持久闩锁，不引入数据库：

| 提交点 | 持久状态 | 自动恢复语义 |
|---|---|---|
| 卡密已预占 | `reserved / reserved` | 可只读重建上下文，不产生副作用 |
| 发货点击前 | `reserved / shipping-attempted` | 重启后只读对账；不得再次点击发货 |
| 已确认订单发货 | `reserved / shipped-confirmed` | 可继续准备一次消息；须重新绑定聊天页 |
| 消息点击前 | `reserved / delivery-attempted` | 重启后只读对账；不得再次发送 |
| 消息与订单均确认 | `sent / delivery-confirmed` | 幂等终态，不再执行 |
| 任一不明确或人工介入 | `manual / <bounded-reason>` | 自动链路停止，迟到回执不得解除闩锁 |

每次重要写入必须写后回读。若写入响应丢失，先按 `card_id/order_id` 只读对账；无法唯一确认即 `manual`。单进程仍需测试并发 run，同订单只能获得一个 intent，不同订单争抢最后一张卡只能有一个成功；部署必须机械断言单副本。

## 7. Harness 与 E2E

### 7.1 自动验证层次

1. 单元测试：飞书状态迁移、订单/消息 intent 状态机、幂等、错误脱敏。
2. 服务端集成：预授权必须先于库存访问；订单状态未明确完成不得准备或签发卡密消息；结果不明确进入人工闩锁。
3. AI Harness：讲解、引导、工具触发、HITL、拒答五维度每场景至少 3 跑；执行记录必须列出闲鱼 pack/version、场景 ID 与五维映射、闲鱼子集通过数/总数和报告路径，闲鱼子集必须全绿，不能用其它 pack 通过率代替。
4. 浏览器 E2E：真实扩展 + 受控闲鱼夹具，覆盖订单筛选、聊天发送、回执、发货确认、状态复核。
5. 生产冒烟：仅健康检查、快照装载与无副作用页面识别；真实低价值订单需用户单独授权。

交付状态分为：

- `engineering-verified`：自动测试与受控夹具通过。
- `deployed-disabled`：生产已部署，自动扫描关闭，完成无副作用冒烟。
- `production-accepted`：用户明确授权的一笔低价值待发货订单完成全链路。

前两项不得表述为“首期自动履约闭环完成”；只有 `production-accepted` 才允许启用对应账号的自动扫描。

### 7.2 必测故障

- 待发货计数为零。
- 买家文本伪造“已付款”。
- 订单号与状态不在同一订单块。
- 飞书库存不足、查询失败、写入失败、写后回读不一致。
- 消息发送超时、回执未增加、回执增加超过 1。
- 订单状态成功后、进入消息页前切页或页面实例变化。
- 发货确认弹窗结构变化、验证码、登录失效。
- 发货点击成功但订单状态未变化。
- Service Worker 在任一步骤间重启。
- 重复 alarm、重复 runId、重复 tool call 与迟到回执。
- 飞书预占成功但响应丢失或进程崩溃。
- `shipping-attempted` 写入后点击前崩溃，以及点击后状态快照丢失。
- 订单已发货但消息尚未签发时重启。
- `delivery-attempted` 写入后点击前崩溃，以及消息已发送但回执/SSE 丢失。
- 两个副作用均确认但飞书 `sent` 回填失败。
- 人工处理后迟到结果到达，闩锁仍不被自动解除。
- 同订单并发 run、不同订单并发争抢最后一张卡。
- 卡密 canary 不得出现在 LLM 上下文、SSE、快照、错误、审计、日志或截图。

故障按副作用提交点分类：副作用前的确定失败只能按显式次数做有界重试；签发后能机械证明未执行的调用可重新准备新上下文，但不能复用 nonce；任何可能已执行的结果永久禁止自动重试并进入人工对账；SW/SSE 恢复只允许恢复观察与对账，不得恢复点击动作。

## 8. 阶段、提交与独立复审

### Phase 0：产品定型与计划

- 分支改为 `codex/zen-commerce-agent`。
- 本文成为首期实施锚点。
- 验收：范围、成功语义、延期锚点与生产边界无歧义。

### Phase 1：生产快照与飞书收口

- 闲鱼 pack 提升为正式 `assets` 快照。
- 产品展示与部署说明电商化。
- 飞书运维规则入仓；镜像具备明确的 `lark-cli` 运行依赖。
- 验收：Assembly 加载生产快照；功能评测、构建、单测通过。
- Phase 1 实现时新增唯一聚合命令 `pnpm verify:phase1`，固定执行 dependency lint、全仓 build、串行单测、全量 eval 与生产快照 Assembly 定向测试；阶段执行记录附完整子命令与报告路径，不允许人工选择子集替代。

### Phase 2：订单发货状态闭环

- 只读侦察真实发货页面至不可逆确认前；完整动作与终态只由受控夹具覆盖，真实终态留给用户授权验收。
- 实现服务端可信派生、固定动作、状态复核和人工闩锁。
- 验收：受控夹具完成消息与订单双证据闭环；可能已经发生副作用的异常不重试，副作用前确定失败按 §7.2 分类处理。
- 验收状态：达到 `engineering-verified`，不能替代真实订单验收。
- Phase 2 实现时新增唯一聚合命令 `pnpm verify:phase2`，固定执行 contracts、card-inventory、fulfillment、toolgate、server、extension 的明确测试文件闭集与 `git diff --check`；阶段执行记录附被执行的文件清单和计数。

实施口径（2026-07-22）：发货与消息投递使用两个独立的 bounded tool 和两条独立运营策略。飞书最小状态链为 `reserved → shipping-attempted → shipped-confirmed → delivery-attempted → sent`；详情页只允许服务端派生的唯一“发货”单击，点击后必须由同 URL、同页面生命周期的新快照唯一确认“已发货”。当前生产账号待发货为零，未观察到真实确认弹窗，因此本阶段只达到 `engineering-verified`；若真实按钮打开额外弹窗，当前实现会因状态未确认而转人工，不会猜测或追加点击。

### Phase 3：E2E 与加固

- 扩充 Harness、浏览器 E2E、SW 恢复和回归用例。
- 验收：全仓 build/test、依赖 lint、功能评测和浏览器 E2E 全绿。
- 新增唯一闲鱼浏览器入口 `pnpm test:e2e:xianyu`，覆盖真实扩展与受控夹具；执行记录写明场景数和报告路径。最终门为 `pnpm lint:deps`、`pnpm -r build`、`pnpm -r --workspace-concurrency=1 test`、`pnpm eval`、`pnpm test:e2e:xianyu`、`pnpm test:e2e:sidepanel`。

每个 Phase 完成后先提交，再分别由独立子 agent 从架构边界和测试/故障覆盖两个方向审查。两个 reviewer 绑定该 Phase commit hash，结论记录 P0/P1/P2、文件行号和建议；主流程以独立修复 commit 处理问题并重跑受影响验证，再由原 reviewer 或新的独立 reviewer 复核。P0/P1 必须为 0；P2 必须修复，或由用户明确接受并挂具体阶段/事件锚点。审查结论与测试命令记录到阶段执行记录，未闭环不得进入下一 Phase。

### Phase 4：发布

- 构建已提交 HEAD 的 linux/amd64 镜像与 Chrome 插件 zip。
- 以版本目录部署服务端与 `assets` 快照到 `lingm2:/root/zen-agent`，失败同时回滚镜像与快照。
